# URL Shortener

Сервис для сокращения ссылок на Go. Принимает оригинальный URL и возвращает уникальный alias из 10 символов.

## Запуск

```bash
cp .env.example .env
docker compose up --build
```

Сервис будет доступен на `http://localhost:8080`.

## API

### POST /shorten

Создать сокращённую ссылку

```bash
curl -X POST http://localhost:8080/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com/very/long/url"}'
```

```json
{
  "alias": "aB3_xYzQr1",
  "short_url": "http://localhost:8080/aB3_xYzQr1"
}
```

Запрос является идемпотентным - повторная передача того же URL вернёт тот же alias

### GET /{alias}

Перейти по сокращённой ссылке. Возвращает `302 Found` с редиректом на оригинальный URL

```bash
curl -L http://localhost:8080/aB3_xYzQr1
```

### GET /health

```bash
curl http://localhost:8080/health
# {"status":"ok"}
```

## Конфигурация

Полный список с дефолтами - [`.env.example`](.env.example)

`DB_FLAG` - Хранилище (`postgres` / `in-memory`)
`BASE_URL` - Базовый URL сервиса 
`LOG_LEVEL` - Уровень логов (`debug` / `info` / `warn` / `error`)
`HTTP_ADDR` - Адрес HTTP-сервера
`POSTGRES_HOST` - Хост PostgreSQL
`POSTGRES_PORT` - Порт PostgreSQL
`POSTGRES_USER` - Пользователь БД
`POSTGRES_PASSWORD` - Пароль БД                  
`POSTGRES_DB` - База данных
`INMEMORY_TTL` - Время жизни неиспользуемой записи (`0` - без ограничений)
`INMEMORY_CLEANUP_INTERVAL` - Интервал запуска фоновой очистки

Для смены вида хранения данных используется `DB_FLAG` со значениями `postgres` и `in-memory`

## Принятые решения

### Алгоритм генерации alias

Alias длиной 10 символов из алфавита `a-zA-Z0-9_` (63 символа) генерируется через `crypto/rand`.

Пространство возможных значений: 63^10 . Такое значение позволяет не беспокоится за коллизии даже при миллионах запросов.

Использование `crypto/rand` вместо `math/rand`:
- Нет предсказуемых последовательностей - alias нельзя угадать зная предыдущий
- Нет необходимости в seed - работает корректно при конкурентных вызовах

При коллизии alias сервис делает до 5 повторных попыток. Эта защита чисто теоретическая, но делает поведение сервиса предсказуемым при любой нагрузке.

### Два индекса в in-memory хранилище

```
byAlias: alias - URL (поиск по GET /{alias})
byOriginal: original - alias (проверка идемпотентности)
```

Оба поиска выполняются за O(1). Альтернатива - один индекс и линейный перебор при проверке оригинального URL - давала бы O(n) при каждом POST-запросе.

### Идемпотентность

Один оригинальный URL всегда возвращает один и тот же alias. Это поведение заложено в хранилище:

- **In-memory**: при сохранении проверяется `byOriginal` - если запись есть, возвращается существующий alias
- **PostgreSQL**: `INSERT ... ON CONFLICT (original) DO NOTHING RETURNING *` - атомарная операция без race condition между CHECK и INSERT

### Интерфейсы определяются потребителем

В Go интерфейс принадлежит тому, кто его использует, а не тому, кто его реализует. Этот принцип применён на каждом слое:

- `UrlRepo` определён в пакете `service` - сервис описывает что ему нужно от хранилища, пакет `repository` не импортируется
- `urlService` определён в пакете `transport/http/handler` - хендлер описывает что ему нужно от сервиса, пакет `service` не импортируется

Каждый слой зависит только от своих абстракций. Это упрощает тестирование - достаточно реализовать интерфейс в тесте, не поднимая реальных зависимостей.

### Sliding TTL для in-memory хранилища

Без ограничений in-memory хранилище растёт бесконечно. Чтобы этого избежать, реализован механизм sliding expiration: каждое обращение к записи (чтение или запись) сдвигает её время жизни вперёд на `INMEMORY_TTL`. Фоновая горутина раз в `INMEMORY_CLEANUP_INTERVAL` удаляет записи, к которым никто не обращался дольше TTL.

Плюсом данного подхода является то, что популярные ссылки живут вечно, а забытые удаляются сами.

При `INMEMORY_TTL=0` (дефолт) очистка отключена, хранилище работает как раньше.

Для PostgreSQL очистка устаревших записей решается на уровне инфраструктуры - можем добавить крон-задачу с `DELETE FROM urls WHERE created_at < NOW() - INTERVAL` или использовать расширение `pg_cron`.

### Конкурентная безопасность

- **In-memory без TTL**: `sync.RWMutex` - параллельные читатели не блокируют друг друга
- **In-memory с TTL**: каждое чтение обновляет `expiresAt`, поэтому используется write lock - осознанный компромисс между памятью и параллелизмом чтения
- **PostgreSQL**: уникальные индексы на уровне БД + `ON CONFLICT` гарантируют атомарность без дополнительных блокировок на стороне приложения

### Встроенные миграции

SQL-миграции встроены в бинарник через `//go:embed`. При старте сервис автоматически применяет новые миграции - не нужно копировать SQL-файлы в Docker-образ или запускать отдельный init-контейнер.

### Graceful shutdown

При получении `SIGTERM`/`SIGINT` сервер перестаёт принимать новые соединения и ждёт завершения активных запросов до 5 секунд. Для `Shutdown` используется отдельный контекст с таймаутом - сигнальный контекст к этому моменту уже отменён и использовать его нельзя.
